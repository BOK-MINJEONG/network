# IPv6 Transition 환경에서의 Packet Translation과 NAT64

> IPv6 Client는 IPv4 Server와 어떻게 통신하는가?

## 1.	패킷 구조: IPv4와 IPv6 헤더 비교
- IPv4 Packet
  - Source IP
  - Destination IP
  - TTL
  - Protocol
- IPv6 Packet


## 2.	지워지지 않는 IPv4의 현실
-	IPv6는 오래전 등장했지만, 현실의 엔터프라이즈 서버와 레거시 인프라는 여전히 IPv4 기반으로 운영되고 있습니다. 
-	최신 모바일 및 데이터 아키텍처는 IPv6-only로의 극단적인 효율을 꾀하지만, 외부 웹 서비스와의 연결 고리는 생각보다 견고합니다.
-	최신 통계 지표에 따르면 전 세계 활성 웹 서비스의 무려 60% 이상이 여전히 IPv4 전용으로 작동하고 있습니다.
-	특히 고도의 보수성을 요구하는 금융 및 공공 기업 비즈니스 내부 시스템의 경우 무려 70%를 상회하는 비율이 오직 IPv4 주소만을 부여 받아 운영 중입니다. 
-	이 간극을 메우는 것이 바로 **Translation 기술**의 핵심 과제입니다.

## 3.	IPv6 전환을 위한 주요 전략들
1. **Dual Stack**
    -	IPv4와 IPv6를 동시에 운영. 가장 안정적이지만 리소스 소모가 큼
2. **Tunneling**
    -	IPv6 패킷을 IPv4 패킷 내부에 캡슐화. 네트워크 경계 연결에 유리.
3. **Translation**
    -	헤더를 직접 재작성(Rewrite). NAT64 기술


## 4.	터널링(Tunneling) vs 변환(Translation)
-	터널링은 출발지와 목적지가 모두 IPv6 패킷을 해석할 수 있는 경우에 한정되어 사용이 가능합니다. 즉, 터널링은 양 끝단이 모두 IPv6를 이해해야 하지만, Translation은 한 쪽이 IPv4-only여도 통신이 가능하게 합니다.
    > 핵심 원리: Header Rewrite   
    > IPv4 Src/Dst -> IPv6 Src/Dst
-	터널링보다 변환이 더 많이 쓰이는 현실적인 이유
    - **서버(콘텐츠) 시장의 보수성 때문**   
    전 세계 이동통신사들은 주소 고갈 문제를 해결하기 위해 스마트폰 망을 IPv6-only로 빠르게 전환했음. 하지만 전 세계의 수많은 웹사이트, API 서버, 스타트업들의 인프라는 여전히 IPv4-only인 경우가 대다수. 
      - 터널링을 쓰려면: 구글, 네이버뿐만 아니라 동네 작은 쇼핑몰 서버까지 전부 IPv6를 이해할 수 있도록 인프라를 갈아 엎거나 터널링 장비를 둬야 함 -> 현실적으로 불가능
      - NAT64를 쓰면: 서버들은 세상이 IPv6로 바뀌든 말든 가만히 있어도 됨. 통신사(망 제공자) 단에서 NAT64 장비 한 대만 두고 싹 다 번역해주면 됨
-	그럼 터널링은 언제 사용?
    - 터널링은 주로 기업 내부 망 연결이나 통신사 간의 백본망 연동처럼, 양 끝단의 장비를 제어할 수 있는 통제된 환경에서 주로 사용
    - 터널링의 한계: 패킷에 껍데기를 계속 씌우다 보니 패킷 크기가 커져서 MTU 이슈가 발생해 패킷이 쪼개지거나 성능 저하가 올 수 있음. 또한, 양 끝 단에서 캡슐화/디캡슐화를 지원해야 하므로 범용 인터넷에는 맞지 않음
    - 따라서 현재 전 세계 이동통신망은 인프라 변경이 어려운 기존 IPv4 서버들을 그대로 포용하면서도, 가입자에게는 IPv6를 서비스하기 위해 터널링보다는 백본 단에서 한 번에 통역을 처리해주는 NAT64/DNS64 변환 기술을 주류로 채택하여 운영하고 있습니다.


## 5.	NAT64와 DNS64
-	이 두 기술의 조합이 오늘날 IPv6 전환의 표준 시나리오
    - **DNS64**: IPv4 A 레코드를 AAAA 레코드로 합성
    - **NAT64**: 실제 패킷의 헤더를 IPv6에서 IPv4로 변환
-	‘서버의 IPv4 주소를 클라이언트는 이해할 수 없다.’
    - IPv6-only client는 도메인 네임 해석 시 오직 AAAA 레코드만을 요청하여 수신합니다. 만약 대상 서버가 IPv4-only라면 DNS는 오직 A 레코드만을 반환하게 되고, 클라이언트는 해당 IP로의 접속 경로 설계가 애초에 불가능해집니다. 이러한 한계를 뛰어넘기 위해 DNS64 장비가 중간에서 개입하여 ‘가상의 IPv6 주소’를 조작하여 전달해 주어야 통신이 시작될 수 있습니다.

## 6.	DNS64: 가상 AAAA 레코드 합성 원리
  1. 원본 레코드 확인 (IPv4-only 판별)
      - Authoritative DNS에 도메인의 IPv6 주소(AAAA)가 없고, IPv4 주소(A 레코드)만 존재함을 확인
  2. IP4 주소의 16진수 변환
      - 확인된 10진수 IPv4 주소(예. 1.2.3.4)를 IPv6 헤더에 삽입할 수 있도록 16진수(0102:0304)로 변환
  3. 고유 Prefix 결합
      - NAT64 장비와 약속된 전용 IPv6 수소 대역인 Well-Known Prefix(64:ff9b::/96)를 선두에 배치
      - IPv6 = Prefix (96) + IPv4 (32)
  4. 가상 AAAA 레코드 완성 및 응답
      - 프리픽스(96비트) 뒤에 16진수 IPv4 주소(32비트)를 이어 붙여 총 128비트의 가상 IPv6 주소를 완성하여 클라이언트에 응답


## 7.	NAT64 패킷 변환 메커니즘
  1. IPv6 인바운드 패킷 수신
      -	클라이언트가 합성된 가상 IPv6 주소를 목적지로 하여 패킷을 발송.    
      > [IPv6 Header]    
      > src: 2001:db8::1 -> dst: 64:ff9b::0102:0304
  2. 가상 주소 및 프리픽스 판별
      -	NAT64 장비가 목적지 IP의 Well-Known Prefix(64:ff9b::/96)을 인식
      -	해당 패킷이 IPv4로 변환되어야 하는 ‘프로토콜 변환 대상’임을 인지하고 프로세스를 시작.
  3. 변환 주소 Pool (IPv4 Pool) 매핑 및 자원 할당
      -	목적지 변환: 가상 IPv6 주소의 하위 32비트에서 실제 서버의 IPv4 주소(1.2.3.4)를 추출
      -	출발지 변환 (Pool 매핑): 서버가 응답을 보낼 수 있도록, NAT64가 보유한 공인 IPv4 변환 주소 Pool에서 가용한 주소(ex. 5.6.7.8)와 임의의 포트 번호를 선택하여 클라이언트의 IPv6 주소와 1:1 매핑.
        - 서버가 답장을 주려면 IPv4 형태의 출발지 주소가 필요한데 클라이언트는 IPv6 주소밖에 없음. 그래서 NAT64 장비는 자기가 관리하는 IPv4 주소 Pool에서 남는 공인 IPv4 주소를 하나 꺼내서 클라이언트의 대리인 주소로 지정해 줌. 이때 장비는 [클라이언트의 IPv6 주소 <-> Pool에서 꺼낸 IPv4 주소 + 포트] 매핑 테이블을 메모리에 기록해 둠. 그래야 나중에 서버가 response를 했을 때 원래 주인인 클라이언트를 찾아줄 수 있기 때문
  4. IPv4 헤더 재작성 (IP Packet 리모델링)
      -	기존 IPv6 헤더를 분리하고, Pool에서 할당받은 IPv4 주소를 출발지로, 추출한 서버 주소를 목적지로 하여 새로운 IPv4 헤더를 조립 후 전송.    
      > [IPv4 Header]    
      > src: 5.6.7.8 -> dst: 1.2.3.4


## 8.	보안 관점에서의 IPv6 Translation 고려사항
-	DNSSEC 우회 위협
    - DNS64 서버가 AAAA 응답 조작 시 서명 검증체계(DNSSEC)가 훼손되어 도메인 스푸핑 및 세션 하이재킹에 노출될 수 있습니다.
-	L3/L4 파싱 DoS 취약점
    - 매 시점 들어오는 변환 패킷의 홉-바이-홉, 엑스텐션 헤더 및 복합 옵션 검증 미흡 시 장비 Crash 망 마비(DoS) 공격이 유효합니다.
-	Stateful 세션 테이블 고갈
    - Stateful 방식 NAT64 세션 관리는 대량의 악성 동시 접속(DDoS) 발생 시 장비 리소스 세션 고갈 유발이 치명적입니다.
